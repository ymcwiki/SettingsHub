# 系统性代码审查记录(2026-07-18)

五维并行深审(codex gpt-5.6-sol,xhigh 档),维度=正确性 / 安全健壮性 / 可用性 / 性能 / 跨版本兼容。原始分维报告共报 153 条,下面是 Claude 跨维度去重、逐条对代码复核后的结论。

复核标记:**已证实**=我读过对应代码行确认属实;**存疑**=与代码一致但未完整追踪触发路径;**严重度存疑**=事实属实但分级取决于策略口径。

---

## P0(数据损坏或红线违反,应最先修)

### 1. 全量还原/卸载还原无条件清 baseline,失败或排队项永久失去原值 — 已证实
`Core/Engine.lua:177-178`。`RestoreAll` 循环里把 `queued` 也计成功,循环后无条件 `wipe(baseline)` 和 `wipe(profile.cvar)`。某项还原失败(降级后键变只读/被移除)或战斗中只排队未写,原值和期望态被抹掉,之后无法重试。四个维度独立命中同一处(correctness / compat C01 / usability U-03 / security #9)。
修法:仅在单键返回 `applied` 后删对应 baseline/desired;`queued` 保留到完成回调,`failed` 保留供重试,全成功才整体清表。

### 2. 诊断与自测实际写用户配置,违反「绝不能改配置」红线 — 已证实
- `SelfTest.lua:255` `/sh diag` 的可写性扫描把当前值 `SetCVar` 原样写回。同值写仍触发 CVAR_UPDATE、服务器同步、其他插件回调。这是 v0.4.1「diag 彻底只读」承诺的实际漏洞。
- `SelfTest.lua:44-61` `/sh test` 真实写值、撤销、回默认再写回;`103-106` `/sh test relog` 把服务器存储项改值保持到重登。
- `SelfTest.lua:326-328` `/sh probe` 对真实勾选框连点两次,走完整写管线,留下 baseline 与撤销日志。
用户红线原话是「一定不能乱改我的配置」。同值写、双击回环都属于写。四维命中(correctness / compat C02-C04 / usability U-01/U-02 / security #16/#17)。
修法:诊断与自测只做只读检查(GetCVarInfo 读只读位判可写性,不写回);probe 只输出焦点链与几何;写回环移到无头桩。

### 3. 导入串无解压上限、无结构校验 — 已证实
`Core/Profiles.lua:160-163`。Decode 只检查 `payload` 是 table 且 `v==1`,无解压字节上限、无 schema。构造高压缩比 deflate bomb 会在 UI 回调里大量分配卡死主线程;合法封装的 `{v=1}`(缺 data)会在预览 `DiffAgainstCurrent` 直接 Lua 报错;部分畸形结构还会在报错前先改了键位/宏/EditMode,形成半导入。这是唯一的外部不可信输入面(分享串),优先级高(security #1/#2 / compat C06 / correctness)。
修法:解压前限串长、用带硬上限的解压、反序列化设对象预算;Decode 前递归校验 data/domains/各域类型与条数;导入先生成不可变计划,全量校验通过再提交。

### 4. console 命令重放无白名单,损坏 SV / 导入串可执行任意 console 命令 — 已证实
`Adapters/ConsoleExec.lua:14` 直接 `ConsoleExec(key.." "..value)` 且无条件 return true;`Core/Profiles.lua:197-198` 导入对 payload 里任意键调 `Engine:Set("consoleexec",...)`;登录 `ReplayAll` 每次重放。恶意/损坏的分享串能在 `profile.consoleexec` 塞任意命令名与参数,每次登录执行。console 命令不能跑 Lua/shell,但能改大量客户端设置。返回值被丢也导致失败命令记成功、永久重放(security #3 / compat C07)。
修法:只允许静态 control 注册表生成的命令名与枚举参数,导入与加载都过同一白名单;`ConsoleExec` 返回非 true 即 `failed`,失败项不进期望态。

### 5. 试穿还原先删恢复表再写回,任一失败永久留下试穿值 — 已证实
`Core/Trial.lua:54-67`。先 `global.trial = nil` 再逐项 `Engine:Set`,失败只是不计数、不回写 SV。注释承诺崩溃兜底,但还原本身不是可恢复事务;某项被拒后下次登录已无 revert 记录。战斗中到期或登出时 secure 项只进内存队列,记录却已清(correctness / security #10)。
修法:逐项 applied 后才从 revert 删该键,failed/queued 继续持久化,整表空才结束 trial。

## P1(确定的功能错误)

### 6. 战斗队列先清空再应用、丢返回值 — 已证实
`Core/CombatQueue.lua:15-29`。Flush 前把队列换空表,忽略 `Engine:Set` 返回值,无条件计 applied 并打印成功。脱战后真失败的项消失且虚报成功。战斗中 Undo/ResetToDefault 返回 queued 时不执行 `undone`/desired 提交,脱战写回后语义不补做,导致当前值、撤销日志、登录重放三者矛盾(correctness / compat C09 / security #11)。

### 7. profile 切换/导入对 bulk 域绕过战斗队列 — 已证实
`Core/Profiles.lua:47-51`。binding/macro/editmode/clickbinding/chatwindow 的 Restore 战斗中返回 `false,"in-combat"`,但调用不检查返回值、不进队列,`Switch` 仍打印已切换。战斗中自动或手动切 profile 得到「CVar 已切/排队、整域仍旧值」的混合态,脱战不补(correctness / compat C08 / usability U-05 / security #12)。

### 8. 导入预览说未知项跳过,应用却仍写入 — 已证实
`Core/Profiles.lua:193-195` 对每个键无条件 `Engine:Set`,而预览把读不到的计为 unknown 并显示「will be skipped」。12.1→12.0.7 迁移时未知项进失败日志而非跳过(compat C05 / usability U-06 / security #5)。

### 9. 导入的 bulk 域只写游戏、不存进 profile — 已证实
`Core/Profiles.lua:204-208` 只 `LogBulk+Restore`,不写 `profile[d]` 也不消费 `payload.domains`。导入键位/宏等当下生效,切走再切回丢失,立即再导出也丢(correctness / usability U-07)。

### 10. 整包撤销忽略逐项失败,仍标整条已撤销;profile 切换 bulk 撤销改坏目标 profile — 已证实
`Adapters/Cvar.lua:30-33` 的 Restore 用 source="import"(属期望态源),bulk 撤销会把旧实机值写进当前 profile 的 desired,持久改坏目标 profile 定义;`Core/Engine.lua:102-107` 只在 Adapter 返回字面 false 才算失败,全 failed/queued 也落 `undone=true`,UI 隐藏再撤销入口。

### 11. profile 域勾选框没真正门控 CVar/console/静音域 — 存疑
`UI/ProfilePage.lua:111` 文案称「只应用/导出勾选的域」,但 `Core/Profiles.lua:45-46/81-86/144-145` 对 CVar、console、静音无条件应用/导出。勾选语义在不同域不一致(usability U-04)。

### 12. onLeave=restore 只切回稀疏 base,不恢复进场景前的值 — 已证实
`Core/Profiles.lua:114-125`。离开只 `Switch(base)`,base 未收录的键保留场景值,「自动回落」只改 profile 名(correctness)。

### 13. 冲突记录是 global 且不带 profile 身份 — 已证实
`Core/Conflicts.lua:14-27`。冲突在 profile A 累计,用户在 B 点「停止管理」删的是 B 的 desired、清的是全局计数,A 仍在管理;多 profile 登录还合并计数(correctness)。

### 14. `_selfWriting` 无 finally,Apply 抛错后整会话卡 true — 已证实
`Core/Engine.lua` Apply 前置 true、正常返回才清。损坏输入让 adapter 或 WoW API 抛错时,标志整会话保持 true,Blame 不再记录外部写入(security #14)。OnEvent 虽有 Guard 包裹,但 Apply 内部没 xpcall。

### 15. EditMode 枚举 12.0.7↔12.1 漂移,无版本门禁,nil 转换静默跳过却报成功 — 存疑
`Adapters/EditMode.lua:42-69`。客户端源显示 system 从 25 项变 27 项、多处枚举改名/移位;导入不据 `game` 字段阻断,转换失败静默跳过仍 return true(compat C12,codex 亦标疑似,C++ 转换器是否自带版本映射无法从 Lua 侧确认)。

### 16. 日志页批量还原 O(k×500) 可能卡死 — 存疑
`Core/Engine.lua:155-176` + `UI/LogPage.lua:226-239`。全量还原对每个键逐条 Set,每次 Notify 触发日志页重建最多 500 条撤销环与冲突列表、新建 DataProvider。管理上千键时是上千次整页重建(performance #1,无实机 profile 前标疑似)。

## P2(边界退化 / 策略性)

### 17. taint:写 StaticPopupDialogs(11 处)、UISpecialFrames(2 处)、超出 3 个约定的命名全局 — 严重度存疑
事实属实(`UI/*` 与 `SelfTest.lua`)。违反 SPEC 零-taint 字面红线。但 StaticPopupDialogs 是暴雪认可的标准弹窗写法,当前客户端正常工作。correctness 报 P0、security 报 P2,分歧就在此。我的判断:是真实 SPEC 越界,但不是活的崩溃风险,按「要么改插件私有弹窗、要么在 SPEC 里立为经 taintLog 验证的显式例外」处理,不与前五条抢工期。

### 18. SavedVariables 骨架 / 撤销环 head 未校验就交 AceDB — 存疑
`Core/Bootstrap.lua:52`、`Core/Engine.lua:19`。手工改坏的 SV(字段类型错、head 越界)可能在初始化各事件里报错。需要人为损坏 SV,概率低,且 OnEvent 现有 Guard 会记录(security #7/#8)。

### 19. 登录重放写任意类型 desired 值的 tostring — 存疑
`Core/Replay.lua` + `Adapters/Cvar.lua:15`。desired 被损坏 SV 改成 table/boolean 时会写成 `table: 0x...`(security #4),同属损坏-SV 类。

### 20-25. 性能高频路径 — 已证实(方向性,非实机帧耗)
每次开窗 wipe 重造全量枚举约 1650 次 GetCVarInfo(`Core/Enum.lua`);每次开窗全量重建搜索 blob(`UI/Search.lua:42-67`);搜索每字符全表扫描+换 Provider,跨页首字符扫两次;单项写入后仍全量类别统计+查询+重建 Provider;Engine 监听器扇出+当前页双刷新;登录与每次进世界全扫期望态;百科 258KB+拼音表无条件常驻(即使从不开窗)。均可增量化。

### 26. deflate 尾随字节不拒绝 — 已证实
`Core/Profiles.lua:160` 只接 DecompressDeflate 第一个返回值,尾部多余字节被静默忽略(security #6)。

### 27. `/sh` 别名无占用检查 — 已证实
`Core/Bootstrap.lua:80-82` 无条件设 `SLASH_SETTINGSHUB2="/sh"`,违反 SPEC「已占用则放弃别名」(compat C13,轻微)。

## 建议修复批次

1. **红线与数据损坏批**(1、2、5、10 的 profile 改坏部分):最高优先,改动集中在 Engine.RestoreAll / SelfTest / Trial / Cvar.Restore,可小步外科手术,配无头断言。
2. **导入健壮性批**(3、4、8、26):Decode 加 schema 校验 + console 白名单 + 未知项真跳过 + 解压硬上限,一次把外部输入面收口。
3. **战斗队列事务批**(6、7、9、14):把排队/整域 Restore 建成带完成回调的事务,统一 applied/queued/failed。
4. **性能批**(16、20-25):枚举/搜索 blob 增量化,写入定向刷新,百科改按需加载。
5. **策略批**(17、27):taint 例外治理与别名占用检查,与用户确认 SPEC 口径后处理。

前两批是安全与红线相关,建议立即做;后三批可排期。
